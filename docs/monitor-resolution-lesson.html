CC:Tweaked · Monitor Reference

What is one pixel?

A hands-on datasheet for the slot machine's medium. Three grids stack inside every monitor — cells, subpixels, real pixels. Once you can read all three, "move it up one pixel" stops being a guess.

Everything the slot draws lands on a grid of character cells — the same grid a text terminal uses. monitor.getSize() reports that grid in cells, not pixels. The pixel-art look comes from a trick played inside each cell. To steer layout, you need the exact size of a cell and of the trick's sub-unit. Both are fixed numbers. Here they are, verified:

Figure 01 — Anatomy of one cell

Six by nine, split two by three

A single cell is a fixed 6-wide × 9-tall block of real pixels. The teletext characters (codes 128–159) carve it into a 2 × 3 grid of subpixels — each subpixel is a 3 × 3 patch. The catch: a cell can show only two colours at once (a foreground and a background). Every subpixel picks one of those two.

Figure 02 — The answer

"Move it up one pixel" — one what?

In subpixel.lua a "pixel" is one subpixel. So one step is 3 real pixels — which is ½ a cell across and ⅓ a cell down. Hover the grid: the readout shows the same point in all three coordinate systems at once. The canvas is cols·2 wide and rows·3 tall.

Figure 03 — The teletext trick

How six subpixels become one character

The monitor can't set a subpixel directly — it can only place a character with a foreground and background colour. Codes 128–159 are the 32 shapes that fill a 2×3 pattern. Two colours per cell is the whole budget. Diagonal detail or a third colour inside one cell is impossible — that's the "subpixel weirdness".

Figure 04 — The two-colour collapse

What happens when you ask for three colours

Paint the 2×3 block with as many colours as you like, then run the encoder. It does exactly what encodeCell does: the most common colour becomes the background, the first different one becomes the foreground, and everything else snaps to whichever of those two it is not. This is why a gradient can look muddy where colours meet — each cell rounds to two.

Figure 05 — Text scale ↔ resolution

Smaller scale, more cells, finer canvas

The physical monitor is fixed; setTextScale only changes how big each cell is drawn. Smaller scale ⇒ more cells fit ⇒ the subpixel canvas gets bigger. The slot runs at 0.5 — the maximum resolution. The exact cell count for a given block arrangement is whatever getSize() returns in-world; slide below to see how it drives the derived canvas and real-pixel dimensions.

Figure 06 — The slot machine, dissected

Two coordinate systems on one screen

Here is the payoff. The slot draws its body — gradient, reel viewport, frame bars, bulbs, banner — on the subpixel canvas, using the layout math in topLayout(). But the economy header (Alice 240 MB) and the WIN!/LOSE label are plain write calls on the cell grid, laid on top. Same monitor, two grids. Toggle the layers to see the seam.
